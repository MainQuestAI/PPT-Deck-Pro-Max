# Illustrative Data Guide

## 目标

当原稿描述了一个概念性缺口、趋势或对比，但没有给出具体数字时，AI 应主动推断合理的说明性数据，用于增强视觉冲击力。

**核心原则：一个 25% 的仪表盘比"理解不足"四个字有 10 倍的冲击力。说明性数据的价值不在精确，在于让概念变成可视化的锚点。**

## 什么时候生成说明性数据

### 应该生成

1. 内容描述了"缺口/断裂/不足"但没有量化 → 推断百分比
2. 内容做了"A vs B"对比但没有量化差异 → 推断指数（如 140 idx vs 100 idx）
3. 内容描述了"效率提升/成本降低"但没有具体值 → 推断改善幅度
4. 内容描述了"覆盖率/完成度"但没有百分比 → 推断当前水平

### 不应该生成

1. 原稿已有具体数字（如"6.02 亿"）→ 直接使用，标记 factual
2. 法律/合规/医疗相关的数据 → 不能编造
3. 客户名称、品牌案例的具体结果 → 不能编造

## 如何推断合理值

### 缺口/断裂类

描述程度越严重，数值应越低：

- "严重不足/几乎没有" → 10%-20%
- "明显不足/差距很大" → 20%-35%
- "部分不足/有待提升" → 35%-50%
- "基本具备/仍有空间" → 50%-70%

### 对比类（索引/指数）

以传统方式为基准 100，AI 方式的改善幅度：

- "显著提升" → 130-150 idx
- "明显改善" → 115-130 idx
- "有所改善" → 105-115 idx
- "降低成本/减少浪费" → 60-80 idx（越低越好）

### 趋势类

- "快速增长" → 斜率陡峭的上升线
- "稳步提升" → 平缓上升线
- "两种方案对比" → 一条上升线（AI）+ 一条平缓线（传统）

## 来源级别（5 级）

| 级别 | 含义 | 可信度 | 示例 |
|------|------|--------|------|
| `expert_confirmed` | 专家直接提供或确认的数据 | 最高 | "活跃人群包 < 1%" — 来自客户经验 |
| `factual` | 原稿中有明确来源的真实数据 | 高 | "CNNIC: 6.02 亿" |
| `industry` | 行业公开数据，有出处但非第一手 | 中高 | "QuestMobile: 3.38 亿" |
| `inferred` | AI 基于上下文推断的说明性数据 | 中 | "深层意图理解率约 25%" |
| `decorative` | 纯为视觉效果，无业务含义 | 最低 | 柱状图的相对高度 |

Compression 优先级：expert_confirmed > factual > industry > inferred > decorative

Expert Mode 下的升级路径：
- 专家提供真实数字 → `expert_confirmed`
- 专家确认 AI 推断合理 → `expert_confirmed`（原值保留，标注"经专家确认"）
- 专家未覆盖 → 保持 `inferred`，标记 `ai_inferred`

## 标记规范

在 `deck_visual_composition.md` 中：

```
指标：深层意图理解率 | value=25% | type=gauge | illustrative=true
指标：首购转化率提升 | value=140idx | type=bar | illustrative=true
指标：AI 用户规模 | value=6.02亿 | type=metric | illustrative=false
```

Build 时的处理：

- `illustrative=true` 的数据，可选在图表下方添加微小的"*示意数据"标注
- `illustrative=false` 的数据，正常展示
- 用户后续提供真实数据时，替换对应条目并改为 `illustrative=false`

## 说明性数据的审美原则

1. **不要太精确**——写"~25%"比"23.7%"更诚实，也更有设计感
2. **不要太圆整**——写"25%"比"20%"更有说服力（看起来像算过的）
3. **三个并列数据不要一样**——25% / 20% / 30% 比 25% / 25% / 25% 更真实
4. **对比指数不要太夸张**——140 idx 比 300 idx 更可信
