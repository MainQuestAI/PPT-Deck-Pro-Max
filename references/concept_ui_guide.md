# Concept UI Guide

## 目标

当真实产品截图不可用时，不要放空占位符。用概念化 UI skeleton（仿产品控制台骨架）建立"这个系统已经存在"的感受。

**核心原则：空占位符说的是"这个东西还没做完"。概念化 UI 说的是"我们已经想清楚了这个系统长什么样"。两者给读者的信心完全不同。**

## 什么是概念化 UI

概念化 UI 不是伪造产品截图。它是用简单的 shape/色块/标签组合，表达"如果这个功能存在，它的界面大致长这样"。

### 示例

场景页说"首购转化 Runtime"，概念化 UI 可以是：

```
┌─ ● ● ●  Private-domain conversion runtime ──────────┐
│                                                       │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐  │
│  │ 用户状态    │  │ 障碍判断    │  │ 推荐动作    │  │
│  │ ▓▓▓░░░░     │  │ 信任不足    │  │ 推成分背书  │  │
│  └─────────────┘  └─────────────┘  └─────────────┘  │
│                                                       │
│  ┌──────────────────────────────────────────────────┐│
│  │ 转化漏斗   ▓▓▓▓▓▓▓▓▓▓▓░░░░░                     ││
│  └──────────────────────────────────────────────────┘│
└───────────────────────────────────────────────────────┘
```

这不是真实产品界面，但它让读者感受到"这个系统有控制台、有状态面板、有漏斗视图"。

## 何时使用概念化 UI

| 场景 | 应该用 |
|------|-------|
| 产品截图不可用（还没有真实系统） | 概念化 UI skeleton |
| 产品截图可用但用户选择先跳过 | 概念化 UI（后续替换为真实截图） |
| 页面需要"产品存在感"但没有对应功能 | 概念化 UI |
| 纯方法论页面（不涉及产品界面） | 不需要 UI，用图表/架构图 |

## 概念化 UI 的构成元素

用 Build AI 能实现的基础元素组合：

1. **窗口框架** — 圆角矩形 + 左上三个彩色小圆点（macOS 风格）+ 标题栏文字
2. **面板/卡片** — 内部分区的色块，代表不同的功能模块
3. **状态指示** — 进度条、状态标签（active/pending/blocked）
4. **数据骨架** — 灰色线条代表数据行，模拟列表或表格
5. **图表骨架** — 简单的柱状/折线轮廓，不需要真实数据
6. **标签文字** — 用业务术语（不是 Lorem ipsum）标注每个区域

## 概念化 UI 的命名规范

在 `deck_visual_composition.md` 中声明：

```
### 概念化 UI
类型：concept_ui
标题：Private-domain conversion runtime
面板：用户状态 | 障碍判断 | 推荐动作
底部：转化漏斗进度条
风格：terminal_window（深色底 + 彩色圆点窗口栏）
```

Build AI 看到这个声明后，应渲染一个完整的窗口框架 + 内部面板，而不是放一个"SCREENSHOT PLACEHOLDER"矩形。

## 禁止事项

1. **禁止空占位符** — 不允许出现"SCREENSHOT PLACEHOLDER"或空色块
2. **禁止伪造真实数据** — 概念化 UI 中的数据骨架应该是灰色线条或模糊色块，不要写具体数字冒充真实系统
3. **禁止过度复杂** — 概念化 UI 是骨架，不是 Figma 原型。5-8 个元素足够
4. **禁止和实际产品风格冲突** — 如果已有产品截图的页面用了某种配色，概念化 UI 应保持一致

## 世界观贡献

概念化 UI 的根本价值不是"填充空白"，而是**参与构建整套 deck 的世界观闭合度**。

当每个场景页都有一个控制台窗口，读者会从第 8 页开始产生一个感觉：

> "这不是在讲一个概念，这是在展示一套已经被想清楚的系统。"

这个感觉就是"一眼可用"的来源。
