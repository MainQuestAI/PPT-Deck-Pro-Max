# Asset Pipeline Guide

## 目标

让 Deck 中的 proof 页和关键页能嵌入真实的产品截图，而不是只用抽象图表或文字描述。

## 何时需要截图 vs 何时用图表

需要截图的场景：

1. 样例页 — 展示"产品真的能跑"
2. 功能展示页 — 展示"这个功能长什么样"
3. 前后对比页 — 展示"用了之后变成什么样"
4. 数据看板页 — 展示"可以看到什么指标"

用图表即可的场景：

1. 架构/系统页 — 展示逻辑关系
2. 流程/阶段页 — 展示步骤
3. 对比/差异页 — 展示抽象对比
4. 价值/优先级页 — 展示判断

## 配图需求梳理流程

### 时机

在 Step 5（Editorial Compression）完成后、Step 6（Visual System）之前。

### AI 的职责

1. 扫描 `deck_clean_pages.md` 的每一页
2. 识别 proof beat、hero_proof、hero_system 等需要真实截图的页
3. 为每页写出"需要什么截图"的描述
4. 输出 `deck_asset_plan.md`
5. 暂停，向用户确认获取方式

### 向用户请求素材的话术模板

```
我分析了这套 Deck 的内容，以下页面需要真实产品截图来强化说服力：

第 5 页（样例页）：需要产品主仪表盘的全景截图
第 6 页（系统页）：需要核心功能界面截图
第 7 页（对比页）：需要优化前后的对比截图

请提供以下任一方式：
1. 产品的访问 URL（我可以自动截图）
2. 直接提供截图文件
3. 先跳过（我会用占位符，后续再替换）

如果提供 URL，是否需要登录？如果需要，请提供 cookies 文件或在浏览器中导出。
```

## deck_asset_plan.md 格式

```markdown
# Asset Plan

## 配图需求总览

| 页码 | Beat | 需要什么 | 优先级 | 获取方式 |
|------|------|---------|--------|---------|
| 第 5 页 | proof | 产品仪表盘全景 | 高 | 待确认 |
| 第 6 页 | resolution | 核心功能界面 | 中 | 待确认 |

## 逐页配图说明

### 第 5 页 — 产品仪表盘

- 需要什么：仪表盘全景，包含主要图表和筛选面板
- 设备壳：macbook
- 位置建议：right（左侧放描述文案，右侧放截图）
- URL：（待用户提供）

### 第 6 页 — 核心功能

- 需要什么：功能界面截图，展示核心操作流程
- 设备壳：browser
- 位置建议：center
- URL：（待用户提供）
```

## 设备壳选择策略

| 产品类型 | 推荐设备壳 |
|---------|-----------|
| SaaS / Web 应用 | macbook 或 browser |
| 移动端应用 | iphone |
| 平板端应用 | tablet |
| CLI / 开发者工具 | terminal |
| 照片 / 实物 | none |

## 占位符规则

1. 当截图不可用时，生成品牌色占位符（使用 theme 的 accent 色）
2. 占位符中心显示配图描述文字
3. 占位符尺寸与最终截图一致
4. 占位符不阻塞 deck 构建流程
5. 后续获得真实截图后，重新执行 `capture-assets` + `apply-mockups` 即可替换
