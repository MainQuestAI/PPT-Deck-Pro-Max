# Compression Rules

## 目标

把长文档翻译成”独立可读的视觉论证”——读者不需要演讲者也能被说服。

**核心原则：80% 的商业 Deck 是被”看”的，不是被”讲”的。** 微信传阅、邮件附件、会前预读、会后传阅——在这些场景下，每一页必须自己解释清楚，不能依赖演讲者补充。

## 两种密度模式

### 文档模式（默认）

适用：大多数商业场景——高层传阅、微信分享、会前预读、客户自行浏览。

- 每页 **250-350 字**
- 保留结论 **+** 关键论据 **+** 核心推导
- 每页必须独立可读——读者看完这一页就能理解”为什么是这个结论”
- 正文区和视觉区**平衡**，不是视觉压倒文字

### 演讲模式

适用：发布会、路演、现场 keynote——有演讲者实时补充上下文。

- 每页 **80-150 字**
- 只保留结论句和视觉主角
- 依赖 speaker notes 补充论据
- 视觉占比 60%+

**如果 Brief 未指定模式，默认使用文档模式。**

## 核心规则

1. 一页只保留一个主结论
2. **文档模式下：结论之后必须保留 1-2 句关键论据**——让读者知道”为什么是这个结论”
3. **每个可视化的概念都应标注视觉形式**——不只是”能变图形的不保留文字”，而是”必须指定变成什么图形”
4. 删除内部讨论语气
5. 删除纯粹的重复推导（但保留推进理解的关键推导步骤）
6. 不保留”为了完整而完整”的说明
7. **为缺口/程度类概念生成说明性数据锚点**（参见 `illustrative_data_guide.md`）

## 证据保真原则

可以删掉：

- 形容词
- 连接词
- 重复解释
- 冗长推导

不能删掉：

1. 具体数字
2. 关键客户或品牌名称
3. 关键系统节点名称
4. 案例锚点

如果证据太多：

1. 先拆页
2. 再变图
3. 最后调整层级

不要为了版面好看而丢掉证据。

## 配图声明

需要产品截图或真实素材的页面，应使用 `> 配图:` 声明：

```markdown
> 配图: id=dashboard_main | desc=产品主仪表盘全景 | frame=macbook
> 配图: id=before_after | desc=优化前后对比 | frame=browser | position=right
```

字段说明：

- `id` — 资产唯一标识，对应 `assets/` 目录下的文件名
- `desc` — 描述"需要什么截图"，给 AI 和用户都看
- `frame` — 设备壳类型：macbook / browser / iphone / tablet / terminal / none
- `position` — 在页面中的位置提示：left / right / center / full
- `url` — 可选，自动截图的目标 URL
- `wait` — 可选，等待 CSS 选择器出现后再截
- `viewport` — 可选，截图视口尺寸（默认 1280x800）

proof beat 页和 hero pages 强烈建议声明配图。没有配图的 proof 页只是在"描述产品"，不是"展示产品"。

## 演讲备注

每页可选附带演讲备注。hero pages 的演讲备注是强制的。

格式（在 `deck_clean_pages.md` 的每页区块内）：

```markdown
> 演讲备注: 这一页的核心话术是"传统研究已经跟不上业务节奏"，重点强调时间成本和决策延迟。
```

演讲备注的内容应该是：

1. 这一页的核心话术或开场白
2. 需要重点强调的数字或案例
3. 预期的听众反应和过渡到下一页的衔接话术

演讲备注不是页面文案的重复。它是"站在台上该怎么说"的指引。
